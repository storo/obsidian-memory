# memory-write-rules — Zettelkasten discipline for the vault

Every file written to the Obsidian vault must follow these rules. They exist so notes remain atomic, linkable, and queryable years later.

## Frontmatter (mandatory)

Every `.md` file in the vault needs YAML frontmatter at the top. Minimum schema:

```yaml
---
title: Human-readable title
tags: [tag1, tag2]
created: YYYY-MM-DD
type: permanent | inbox | fleeting | decision | session-log | reference
---
```

Additional fields by type:

- **decision**: add `review_date: YYYY-MM-DD`, `status: active | closed-success | closed-failed`.
- **session-log**: add `updated: YYYY-MM-DD` if edited after creation.
- **permanent**: may add `source` if derived from an external reference.

**Never** write a note without frontmatter. Never.

## Filename rules

- **kebab-case** only: `auth-flow.md`, not `Auth Flow.md` or `authFlow.md`.
- **Date-prefixed** for logs, decisions, inbox: `2026-04-14-<slug>.md`.
- **No spaces, no parentheses, no special chars**. Obsidian can index them but tooling around the vault (grep, Dataview, shell scripts) breaks.
- **Filename matches title slug**. If title is "Use SQLite not Redis for triage queue", file is `use-sqlite-not-redis-for-triage-queue.md`.

## Wikilinks, never markdown links

For **internal** vault references use `[[note-slug]]`, not `[text](path.md)`. Obsidian's graph and backlinks only work with wikilinks.

External (http://) references stay as markdown links — those aren't vault notes.

## Atomicity — one concept per permanent note

A `permanent/` note holds **one idea**. If you find yourself writing "and also" or "in addition", split into two notes linked with wikilinks.

This doesn't apply to:

- `inbox/` — raw capture is allowed to be unstructured.
- `logs/` — session logs intentionally bundle several things.
- `decisions/` — one decision per note, but the note may discuss context, reasoning, and outcome in structured sections.

## Minimum linkage — 2 wikilinks per permanent note

Every permanent note must link to at least 2 other vault notes. If you can't find 2 relations, the note probably isn't ready to be permanent — leave it in `inbox/` or `fleeting/`.

## Folder semantics

- **`permanent/`** — consolidated knowledge, long-lived, densely linked.
- **`inbox/`** — raw capture. Write freely. Curation later.
- **`fleeting/`** — temporary working notes. OK to delete.
- **`logs/`** — session logs (auto-generated by `/obsidian-memory:save` and `SessionEnd` hook). Don't hand-write here unless correcting an auto-log.
- **`decisions/`** — one file per decision, managed by `/obsidian-memory:decide` and `/obsidian-memory:review`.
- **`projects/<name>/`** — project-specific notes. MOCs, architecture, open questions. Use when content is only meaningful in that project's context.
- **`templates/`** — Templater templates. Don't read these as knowledge.

## Tags (not folders) for typed memories

The conceptual types `#user`, `#feedback`, `#project/<slug>`, `#reference` are **tags**, not folders. A feedback memory about a project goes in `permanent/` with `tags: [feedback, project/sapo]` — not in a `feedback/` subfolder.

This is the Zettelkasten choice: organize by topic (via links) and retrieval (via tags/queries), not by hierarchy.

## Things you must never do

- **Delete notes without asking the user.** Ever. Even if they look stale.
- **Edit other notes' frontmatter** when your task is writing a new note. Leave neighbors alone.
- **Write secrets** — tokens, passwords, private keys. If the user dictates one, refuse and explain.
- **Change folder structure** without documenting it in a permanent note and updating the vault `CLAUDE.md`.
- **Write very long notes.** If a note exceeds ~500 lines, split it.
